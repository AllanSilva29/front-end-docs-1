Em qualquer aplicação web, certas partes da interface são consistentes em quase todas as páginas: a barra de navegação no topo, uma sidebar lateral ou um rodapé no final. Repetir esses componentes em cada arquivo de página é ineficiente e difícil de manter.

```jsx
// O que queremos evitar:
function HomePage() {
  return (
    <div>
      <Navbar /> {/* Repetido */}
      <main>Conteúdo da Home</main>
      <Footer /> {/* Repetido */}
    </div>
  );
}

function AboutPage() {
  return (
    <div>
      <Navbar /> {/* Repetido */}
      <main>Conteúdo sobre nós</main>
      <Footer /> {/* Repetido */}
    </div>
  );
}
```

O padrão **Componente de Layout** resolve esse problema criando um "molde" reutilizável para a estrutura da sua página.

## O Padrão de Componente de Layout

A ideia é criar um componente, vamos chamá-lo de `<MainLayout />`, que define a estrutura comum da UI. Este componente usa uma `prop` especial do React: `children`.

A `prop` `children` contém qualquer conteúdo que é passado *entre* as tags de abertura e fechamento de um componente. O componente de layout renderiza essa `prop` em um local específico dentro de sua estrutura.

### Criando o Componente de Layout

<Tabs>
<TabItem label="components/MainLayout.jsx">
```jsx
import Navbar from './Navbar';
import Footer from './Footer';

// O MainLayout recebe 'children' como prop.
function MainLayout({ children }) {
  return (
    <div className="app-container">
      <Navbar />
      <main className="content">
        {/* Aqui é onde o conteúdo específico da página será renderizado */}
        {children}
      </main>
      <Footer />
    </div>
  );
}

export default MainLayout;
```
</TabItem>
</Tabs>

## Integrando o Layout com React Router

Agora, como usamos nosso `<MainLayout />` com o sistema de rotas? A maneira mais limpa e moderna de fazer isso com o React Router v6 é através de **rotas aninhadas**.

A ideia é criar uma "rota de layout" que não tem um caminho (`path`) próprio, mas renderiza o componente de layout. As rotas reais das páginas se tornam filhas dessa rota de layout. Para que isso funcione, o componente de layout precisa renderizar um componente `<Outlet>` do React Router, que atua como um placeholder para o conteúdo da rota filha.

<Tabs>
<TabItem label="1. Atualize o MainLayout.jsx">
```jsx
import { Outlet } from 'react-router-dom'; // Importe o Outlet
import Navbar from './Navbar';
import Footer from './Footer';

function MainLayout() { // Não precisa mais de 'children'
  return (
    <div className="app-container">
      <Navbar />
      <main className="content">
        {/* O Outlet renderizará o 'element' da rota filha correspondente */}
        <Outlet />
      </main>
      <Footer />
    </div>
  );
}

export default MainLayout;
```
</TabItem>
<TabItem label="2. Atualize o App.jsx (Rotas)">
```jsx
import { Routes, Route } from 'react-router-dom';
import MainLayout from './components/MainLayout';
import HomePage from './pages/HomePage';
import AboutPage from './pages/AboutPage';

function App() {
  return (
    <Routes>
      {/* Rota de Layout */}
      <Route element={<MainLayout />}>
        {/* Rotas Filhas */}
        <Route path="/" element={<HomePage />} />
        <Route path="/about" element={<AboutPage />} />
      </Route>
      {/* Você pode ter outras rotas aqui que NÃO usam o MainLayout */}
    </Routes>
  );
}
```
</TabItem>
</Tabs>

<Aside type="tip">
Esta abordagem de rotas aninhadas com `<Outlet />` é extremamente poderosa. Ela permite criar múltiplos layouts (por exemplo, um `<AdminLayout />` para rotas de admin) e compor UIs complexas de forma declarativa e centralizada no seu arquivo de rotas.
</Aside>

## Diferença entre Páginas e Componentes

Com essa estrutura, a distinção entre "páginas" e "componentes" fica mais clara:

-   **Componentes (`/components`):** São os blocos de construção reutilizáveis da sua UI. Eles devem ser agnósticos em relação a rotas e, idealmente, não devem conter lógica de busca de dados complexa. Ex: `<Button>`, `<Card>`, `<Navbar>`.
-   **Páginas (`/pages` ou `/features/.../routes`):** São componentes que são mapeados diretamente para uma rota. Eles são responsáveis por **compor** componentes menores e, geralmente, por orquestrar a busca de dados necessária para aquela visão específica.
